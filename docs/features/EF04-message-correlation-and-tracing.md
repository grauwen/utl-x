# EF04: Message Correlation and Tracing

**Status:** Must-have (hygiene factor — required before go-live)  
**Priority:** Critical  
**Created:** May 2026

---

## Summary

UTLXe does not currently propagate message correlation metadata from input to output. When a message arrives from Azure Service Bus via Dapr, the output message on the output queue has no link back to the input message. In a production scenario with thousands of messages, you cannot trace which input produced which output.

This feature fixes that by implementing the standard messaging triad:

| ID | Stays same across chain? | Value |
|----|:---:|-------|
| **MessageId** | No | New UUID for each message (UTLXe generates one for the output) |
| **CorrelationId** | Yes | Preserved from the original producer — constant across the entire business process |
| **CausationId** | No | The `MessageId` of the input message — links each output to its immediate cause |

All three IDs are UUIDv7 (RFC 9562) — time-ordered, sortable by creation time, with an embedded millisecond timestamp. In a multi-step chain:

```
Producer creates order:      MessageId=UUID-A  CorrelationId=UUID-CORR  CausationId=(none)
UTLXe transforms → invoice:  MessageId=UUID-B  CorrelationId=UUID-CORR  CausationId=UUID-A
Next service → receipt:      MessageId=UUID-C  CorrelationId=UUID-CORR  CausationId=UUID-B
```

The `CorrelationId` links all messages in the same business process. The `CausationId` links each message to its immediate parent — the chain of causation. Together they provide full traceability.

Additionally:
1. W3C Trace Context (OpenTelemetry `traceparent`) is forwarded across the transformation for distributed tracing
2. Custom Service Bus properties are forwarded (configurable)
3. Every message is logged with its MessageId, CorrelationId, and CausationId

## The Problem

### Current state

When UTLXe sends a transformed message to the Dapr output binding, it includes only:

```kotlin
"metadata" to mapOf(
    "source-binding" to bindingName,    // "orders-in"
    "transform-id" to transformId       // "orders-in"
)
```

Missing:
- `MessageId` from the input Service Bus message — not forwarded
- `CorrelationId` from the input message — not forwarded
- `traceparent` / `tracestate` (W3C Trace Context) — Dapr adds it to the inbound call, but UTLXe does not propagate it to the output binding call
- No log entry linking input message to output message

### What this means in production

- Cannot trace which input message produced which output message
- Cannot correlate errors to specific input messages
- Distributed tracing (Azure Monitor, Jaeger, Zipkin) breaks at the UTLXe boundary — the trace span stops when Dapr calls UTLXe and doesn't resume on the output side
- Audit requirements (e.g., Peppol e-invoicing) that demand end-to-end traceability cannot be met

## What Each Layer Provides

### Azure Service Bus message properties

| Property | Type | Purpose |
|----------|------|---------|
| `MessageId` | String | Unique ID for this message (set by producer, UUID if not set) |
| `CorrelationId` | String | Links related messages (request → response, saga steps) |
| `SessionId` | String | Ordered processing within a session |
| `SequenceNumber` | Long | Broker-assigned sequence number (immutable) |
| `EnqueuedTimeUtc` | DateTime | When the message entered the queue |
| Custom properties | Map | Arbitrary key-value pairs (business metadata) |

### What Dapr forwards to UTLXe

When Dapr delivers a Service Bus message to UTLXe via HTTP POST, it includes:

- **HTTP body**: the message payload (data)
- **HTTP headers**:
  - Standard headers (`Content-Type`, etc.)
  - `traceparent` — W3C Trace Context trace ID (added by Dapr automatically)
  - `tracestate` — W3C Trace Context state
- **Metadata** (accessible via Dapr's binding response format): Service Bus properties including `MessageId`, `CorrelationId`, custom properties

### What Dapr accepts on output binding calls

When UTLXe calls `POST localhost:3500/v1.0/bindings/{name}`, the request body is:

```json
{
  "operation": "create",
  "data": "transformed payload",
  "metadata": {
    "MessageId": "output-msg-id",
    "CorrelationId": "preserved-correlation-id",
    "customProperty1": "value1"
  }
}
```

Dapr passes the `metadata` map as Service Bus message properties on the output message. This is the mechanism for preserving correlation.

## UUIDv7

All MessageIds generated by UTLXe use UUIDv7 (RFC 9562). UUIDv7 embeds a millisecond-precision Unix timestamp in the first 48 bits, making IDs sortable by creation time:

```
019078a5-b3c4-7def-8901-234567890abc
^^^^^^^^^^^^^^^^ ^^^^
  timestamp      v7
  (48 bits)      version
```

Benefits over UUIDv4 (random):
- Sortable by creation time — no need to join with a timestamp column
- Timestamp extractable from the ID itself — useful for debugging
- Database index performance — sequential inserts instead of random scatter
- The causation chain UUID-A → UUID-B → UUID-C is naturally time-ordered

**Policy:** UTLXe generates UUIDv7 for all internally created MessageIds. Callers may send any valid UUID (v4, v7, or other) — UTLXe does not reject based on version. UUIDv7 is recommended.

**Kotlin implementation** (no external dependency):

```kotlin
import java.util.UUID
import java.security.SecureRandom

private val random = SecureRandom()

fun uuidV7(): UUID {
    val timestamp = System.currentTimeMillis()
    val msb = (timestamp shl 16) or (0x7000L) or (random.nextLong() and 0x0FFFL)
    val lsb = (0x8000000000000000UL.toLong()) or
              (random.nextLong() and 0x3FFFFFFFFFFFFFFFL)
    return UUID(msb, lsb)
}
```

## Design

### 1. Read incoming metadata

Extract from the Dapr HTTP request:

```kotlin
// W3C Trace Context (OpenTelemetry)
val traceparent = call.request.header("traceparent")
val tracestate = call.request.header("tracestate")

// Service Bus properties forwarded by Dapr
// Dapr forwards Service Bus properties with "metadata." prefix (EF05)
val incomingMessageId = call.request.header("metadata.MessageId")
    ?: call.request.header("MessageId")
val incomingCorrelationId = call.request.header("metadata.CorrelationId")
    ?: call.request.header("CorrelationId")

// All custom properties (for pass-through)
val customProperties = call.request.headers.entries()
    .filter { it.key.startsWith("metadata.") }
    .associate { it.key.removePrefix("metadata.") to it.value.first() }
```

### 2. Propagate to output binding

Extend the output metadata map:

```kotlin
val outputMetadata = mutableMapOf(
    "source-binding" to bindingName,
    "transform-id" to transformId,
)

// The messaging triad
outputMetadata["MessageId"] = UUID.randomUUID().toString()  // new ID for output
if (incomingCorrelationId != null) {
    outputMetadata["CorrelationId"] = incomingCorrelationId  // preserved
}
if (incomingMessageId != null) {
    outputMetadata["CausationId"] = incomingMessageId  // input's MessageId = output's CausationId
}

// Forward custom properties (pass-through)
outputMetadata.putAll(customProperties)
```

The output Dapr binding call becomes:

```json
{
  "operation": "create",
  "data": "transformed payload",
  "metadata": {
    "source-binding": "orders-in",
    "transform-id": "orders-in",
    "CorrelationId": "UUID-CORR",
    "CausationId": "UUID-A",
    "customProperty1": "value1"
  }
}
```

### 3. Forward W3C Trace Context

Propagate `traceparent` and `tracestate` headers on the output binding HTTP call:

```kotlin
val conn = url.openConnection() as HttpURLConnection
conn.requestMethod = "POST"
conn.setRequestProperty("Content-Type", "application/json")

// Forward W3C Trace Context for distributed tracing
if (traceparent != null) conn.setRequestProperty("traceparent", traceparent)
if (tracestate != null) conn.setRequestProperty("tracestate", tracestate)
```

This ensures the OpenTelemetry trace spans the entire flow: producer → Service Bus → Dapr → UTLXe → Dapr → Service Bus → consumer. Azure Monitor, Jaeger, and Zipkin will show the transformation as a span within the distributed trace.

### 4. Direct HTTP (Scenario 3) — request headers

Direct HTTP clients (no Dapr) should send correlation headers:

| Header | Purpose | Required |
|--------|---------|----------|
| `X-Message-Id` | Unique ID for this request | Optional — UTLXe generates a UUID if not provided |
| `X-Correlation-Id` | Links related requests across systems | Optional |
| `traceparent` | W3C Trace Context (OpenTelemetry) | Optional |

Example:

```bash
curl -X POST \
  -H "Content-Type: application/json" \
  -H "X-Message-Id: UUID-A" \
  -H "X-Correlation-Id: UUID-CORR" \
  -d '{"orderId":"12345"}' \
  https://myapp.azurecontainerapps.io/api/transform/invoice-to-ubl
```

If the client sends no headers, UTLXe generates a UUID as the `MessageId` so every request is traceable — even ad-hoc curl calls.

The response includes the correlation headers:

```
HTTP/1.1 200 OK
X-Message-Id: UUID-B
X-Correlation-Id: UUID-CORR
X-Causation-Id: UUID-A
X-Transform-Duration-Ms: 3
```

This gives the client a receipt that can be correlated with logs and monitoring.

### 5. Unified metadata model

Both Dapr and direct HTTP use the same internal correlation model:

| Field | On input message | On output message | Direct HTTP header |
|-------|-----------------|-------------------|-------------------|
| MessageId | Read from input | New UUID (generated by UTLXe) | `X-Message-Id` (or generated) |
| CorrelationId | Read from input | Preserved (same value) | `X-Correlation-Id` (optional) |
| CausationId | Not present (it was set by previous step) | Set to input's `MessageId` | `X-Causation-Id` (optional) |
| traceparent | Read from Dapr | Forwarded | `traceparent` (optional) |

The transformation logic and logging are identical regardless of how the message arrived.

### 6. Log the correlation

Structured log entry for every processed message:

```
INFO  [orders-in] MessageId=UUID-A CorrelationId=UUID-CORR
      CausationId=UUID-PREV → transformed in 3ms
      → output-binding=orders-out output-MessageId=UUID-B
```

And for errors:

```
ERROR [orders-in] MessageId=UUID-A CorrelationId=UUID-CORR
      CausationId=UUID-PREV → FAILED at line 14: Null reference
```

The error ring buffer (EF03) should also store the MessageId and CorrelationId per error entry.

## Configuration

### Default behavior (no configuration needed)

- `CorrelationId`: always forwarded if present in input
- `CausationId`: always set on output (the input's MessageId)
- `traceparent`/`tracestate`: always forwarded if present
- Custom properties: always forwarded (pass-through)
- Structured log: always includes MessageId and CorrelationId

### Optional: disable metadata forwarding

For transformations where custom property pass-through is not desired (e.g., privacy — input properties might contain sensitive data):

```yaml
# transform.yaml
metadataForwarding: none     # don't forward custom properties
```

Options:
- `all` (default) — forward CorrelationId, CausationId, all custom properties
- `correlation` — forward only CorrelationId and CausationId, not custom properties
- `none` — forward nothing (UTLXe still adds source-binding and transform-id)

W3C Trace Context (`traceparent`) is always forwarded regardless of this setting — it is infrastructure, not business data.

## Proto/gRPC changes

The messaging triad must be available on all transports, not just HTTP/Dapr. The proto definition (`utlxe.proto`) has been updated:

**`ExecuteRequest`** — new fields (sequential after existing field 5):
- `message_id` (6) — caller sets, or UTLXe generates UUID
- `causation_id` (7) — MessageId of the message that caused this one
- `metadata` (8) — custom properties to forward
- `traceparent` (9) — W3C Trace Context

**`ExecuteResponse`** — new fields (sequential after existing field 9):
- `message_id` (10) — new UUID for the output message
- `causation_id` (11) — set to input's message_id
- `metadata` (12) — forwarded custom properties

**`BatchItem`** — new fields (sequential after existing field 4): `message_id` (5), `causation_id` (6), `metadata` (7)

**`ExecutePipelineRequest`** — new fields (sequential after existing field 4): `message_id` (5), `causation_id` (6), `metadata` (7), `traceparent` (8)

**`ExecutePipelineResponse`** — new fields (sequential after existing field 10): `message_id` (11), `causation_id` (12), `metadata` (13)

All new fields are sequential and additive — fully backward compatible. Existing wrappers that don't set the new fields still work (empty strings, UTLXe generates UUIDs).

The existing `correlation_id` field (5 on request, 9 on response) remains unchanged — it serves double duty as both the business CorrelationId and the multiplexed response matcher for stdio-proto mode.

### Additional proto fields (from SAP/BizTalk analysis)

The following fields were added to the proto based on cross-host analysis (SAP CPI, BizTalk, Logic Apps, Dapr/CloudEvents). They extend the tracing and metadata story:

**`ExecuteRequest`:**
- `parameters` (10) — transformation input parameters, accessible as `$params.key` in .utlx
- `metadata_forwarding` (11) — controls which metadata keys pass through (`ALL`, `NONE`, `CORRELATION`)
- `tracestate` (12) — W3C Trace Context vendor extensions (companion to `traceparent`)

**`ExecuteResponse`:**
- `error_code` (13) — specific error code (`TRANSFORMATION_NOT_FOUND`, `BUNDLE_NOT_LOADED`, etc.)
- `output_metadata` (14) — rule-emitted business metadata (`OutputMetadata` message: `application_id`, `message_type`, `custom_status`, `custom_identifiers`, `sender`, `receiver`)

**`BatchItem`:**
- `parameters` (8), `traceparent` (9), `tracestate` (10) — per-item trace context and parameters

**`ExecutePipelineRequest`:**
- `parameters` (10), `tracestate` (11)

**`ExecutePipelineResponse`:**
- `error_code` (14), `output_metadata` (15)

**New messages:** `LoadBundleRequest`/`Response`, `UnloadBundleRequest`/`Response`, `OutputMetadata`

**New enums:** `ErrorCode` (10 specific codes), `MetadataForwarding`

See [Proto Reference](../../docs/sdk/proto-reference.md) for the complete field-by-field documentation.

## SDK and wrapper impact

The proto change adds new fields to `ExecuteRequest`, `ExecuteResponse`, `BatchItem`, `ExecutePipelineRequest`, and `ExecutePipelineResponse`. Every SDK and wrapper that uses the proto must be updated.

| SDK / Wrapper | Language | Action required |
|---------------|----------|----------------|
| UTLXe engine | Kotlin/Java | Regenerate proto stubs (`./gradlew generateProto`). Update `GrpcTransport.kt` and `StdioProtoTransport.kt` to read/set new fields. |
| Go wrapper (Open-M) | Go | Regenerate Go stubs from `proto/utlxe/v1/utlxe.proto` (`protoc --go_out`). Update `Execute()` calls to pass `MessageId`, `CorrelationId`, `CausationId`. |
| Python conformance runner | Python | Uses manual protobuf wire format (no generated stubs). Update `engine-runner.py` to include new fields in `ExecuteRequest` serialization. |
| Future .NET SDK | C# | Generate C# stubs from the proto. New fields available from the start. |
| Future Python SDK | Python | Generate Python stubs from the proto. New fields available from the start. |

**Backward compatibility:** The new fields are additive. An old SDK that doesn't set the new fields still works — UTLXe receives empty strings and generates UUIDs. An old SDK receiving a response with the new fields ignores them (standard protobuf forward compatibility). But to *use* tracing, the SDK must be recompiled against the updated proto.

**Recommended SDK update pattern:**

```go
// Go wrapper — before EF04
resp, err := client.Execute(&ExecuteRequest{
    TransformationId: "orders-in",
    Payload:          payload,
    ContentType:      "application/json",
    CorrelationId:    correlationId,
})

// Go wrapper — after EF04 (all IDs are UUIDv7)
resp, err := client.Execute(&ExecuteRequest{
    TransformationId: "orders-in",
    Payload:          payload,
    ContentType:      "application/json",
    CorrelationId:    correlationId,         // UUID-CORR, preserved across chain
    MessageId:        uuid.Must(uuid.NewV7()).String(),  // UUID-A, unique per message
    CausationId:      incomingMessageId,     // UUID-PREV, parent's MessageId
    Traceparent:      incomingTraceparent,   // W3C distributed trace
})

// Response: UTLXe generated a new UUIDv7 for the output
fmt.Printf("Output MessageId: %s\n", resp.MessageId)   // UUID-B (new UUIDv7)
fmt.Printf("CausationId: %s\n", resp.CausationId)       // UUID-A (= input)
```

### C# SDK example (gRPC) — full messaging triad

A complete C# example showing how a future .NET SDK would use the messaging triad. The proto stubs are generated from `proto/utlxe/v1/utlxe.proto`.

```csharp
using Grpc.Net.Client;
using Utlxe.V1;  // generated from utlxe.proto

// ── Setup ──
var channel = GrpcChannel.ForAddress("http://localhost:9090");
var client = new UtlxeService.UtlxeServiceClient(channel);

// ── Load a transformation ──
var loadResp = await client.LoadTransformationAsync(
    new LoadTransformationRequest
    {
        TransformationId = "order-to-invoice",
        UtlxSource = File.ReadAllText("order-to-invoice.utlx"),
        Strategy = "COMPILED"
    });
if (!loadResp.Success)
    throw new Exception($"Load failed: {loadResp.Error}");

// ── Execute with messaging triad (all IDs are UUIDv7) ──
// These come from the upstream system (e.g., Service Bus message properties)
var incomingMessageId = "UUID-A";     // this message's identity
var correlationId     = "UUID-CORR";  // constant across the entire chain
var causationId       = "UUID-PREV";  // what caused this message

var response = await client.ExecuteAsync(new ExecuteRequest
{
    TransformationId = "order-to-invoice",
    Payload = ByteString.CopyFromUtf8(orderJson),
    ContentType = "application/json",

    // Messaging triad — all UUIDv7
    MessageId = incomingMessageId,       // UUID-A
    CorrelationId = correlationId,       // UUID-CORR
    CausationId = causationId,           // UUID-PREV
});

// ── Response: UTLXe generates a new UUIDv7 for the output ──
//
//   Request:                     Response (set by UTLXe):
//     MessageId     = UUID-A  →    MessageId     = UUID-B  (NEW UUIDv7)
//     CorrelationId = UUID-CORR →  CorrelationId = UUID-CORR  (SAME)
//     CausationId   = UUID-PREV →  CausationId   = UUID-A  (= input's MessageId)

Console.WriteLine($"Input  MessageId:   {incomingMessageId}");       // UUID-A
Console.WriteLine($"Output MessageId:   {response.MessageId}");      // UUID-B (new)
Console.WriteLine($"Output CausationId: {response.CausationId}");    // UUID-A
Console.WriteLine($"CorrelationId:      {response.CorrelationId}");  // UUID-CORR

// ── Chain to the next step ──
// The output's MessageId becomes the next step's CausationId

var nextResponse = await client.ExecuteAsync(new ExecuteRequest
{
    TransformationId = "validate-invoice",
    Payload = response.Output,
    ContentType = response.OutputContentType,

    MessageId = Guid.CreateVersion7().ToString(),  // UUID-C (new UUIDv7, .NET 9)
    CorrelationId = response.CorrelationId,        // UUID-CORR — never changes
    CausationId = response.MessageId,              // UUID-B caused this
});

// The causation chain:
//   UUID-PREV → UUID-A → UUID-B → UUID-C
//   All share CorrelationId = UUID-CORR
//   Each CausationId points to its immediate parent
//   All are UUIDv7 — sortable by creation time
```

Key behavior:
- UTLXe **always** sets `response.CausationId = request.MessageId` — automatic, the SDK developer doesn't do it.
- UTLXe **always** generates a new `response.MessageId` (UUID) — the output has its own identity.
- UTLXe **always** echoes `response.CorrelationId = request.CorrelationId` — the saga ID is preserved.
- If the caller doesn't set `request.MessageId`, UTLXe generates one — every message is traceable.

## End-to-end traceability

With EF04, the full correlation chain is:

```
Producer:           MessageId=UUID-A  CorrelationId=UUID-CORR  CausationId=(none)
                         ↓
Service Bus:        stores message with properties
                         ↓
Dapr → UTLXe:       forwards UUID-A, UUID-CORR
                    adds traceparent (W3C Trace Context)
                         ↓
UTLXe log:          INFO [orders-in] MessageId=UUID-A
                    CorrelationId=UUID-CORR → transformed in 3ms
                         ↓
UTLXe → Dapr:       MessageId=UUID-B (new UUIDv7)
                    CorrelationId=UUID-CORR (preserved)
                    CausationId=UUID-A (input's MessageId)
                    traceparent forwarded
                         ↓
Output Service Bus: MessageId=UUID-B
                    CorrelationId=UUID-CORR
                    CausationId=UUID-A
                         ↓
Consumer:           follows CausationId(UUID-A) back to input
                    follows CorrelationId(UUID-CORR) to find entire chain
```

All IDs are UUIDv7 (RFC 9562) — time-ordered with an embedded millisecond timestamp. This means IDs are sortable by creation time, and the timestamp can be extracted from the ID itself.

- UTLXe generates UUIDv7 for all internally created MessageIds (output messages, auto-generated when caller omits).
- Callers may send any valid UUID (v4, v7) — UTLXe does not reject based on version. But v7 is recommended.
- If a caller does not set a `MessageId`, UTLXe generates a UUIDv7.
- The `CorrelationId` is optional — if not set by the producer, there is no correlation across the chain.

Azure Monitor shows the complete distributed trace: producer → Service Bus → Dapr → UTLXe → Dapr → Service Bus → consumer.
