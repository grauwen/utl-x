## 2. Recommendation: Expression Compilation, Not Full Bytecode

**Compile individual expressions to JVM bytecode. Keep the outer execution framework (input parsing, output serialization, multi-input dispatch) interpreted.**

Why:

### Full bytecode (compile everything)
- Would need to generate bytecode for: input parsing, UDM construction, expression evaluation, output serialization
- Input parsing and serialization are already implemented in optimized Java/Kotlin libraries (Jackson, XMLParser, etc.) — regenerating them as bytecode gains nothing
- The AST interpreter overhead is only in the **expression evaluation** — the tree-walking step
- Full bytecode would require generating code for all 652 stdlib functions — massive, error-prone, ongoing maintenance burden
- Any language change (new syntax, new operator) requires updating the bytecode generator

### Expression compilation (compile the hot path only)
- Compile the `.utlx` body expression to a JVM class with a single `execute(inputs: Map<String, UDM>): UDM` method
- Property access (`$input.name`) → direct `UDM.Object.get("name")` calls (no AST node dispatch)
- Function calls (`concat(...)`) → direct static method calls to stdlib functions (no reflective lookup)
- Operators (`>=`, `+`, `==`) → direct comparison/arithmetic (no interpreter switch)
- Control flow (`if/else`, `let`) → JVM bytecode conditionals and locals
- This is where 90%+ of the interpreter overhead lives

### The math

```
Current per-message cost breakdown (TEMPLATE strategy):
  Input parsing:      200μs  (40%)  ← optimized library code, can't improve
  Expression eval:    250μs  (50%)  ← interpreter tree-walking, THIS IS THE TARGET
  Output serialization: 50μs (10%) ← optimized library code, can't improve

With expression compilation:
  Input parsing:      200μs  (55%)  ← unchanged
  Expression eval:     50μs  (14%)  ← 5x faster (direct bytecode, no tree-walking)
  Output serialization: 50μs (14%) ← unchanged
  JIT warmup cost:     ~0μs        ← amortized after first few hundred calls

Total: 300μs → ~300μs first call (compilation at init-time)
       500μs → ~300μs steady state (40% improvement on the hot path)
```

The improvement is real but bounded — input parsing dominates. Combining COMPILED with COPY (pre-built skeleton eliminates parsing) gives the full benefit:

```
COPY + COMPILED:
  Skeleton copy:       10μs  (17%)  ← deep clone pre-built structure
  Fill with data:      30μs  (50%)  ← direct assignment to known nodes
  Expression eval:     15μs  (25%)  ← compiled bytecode, no interpretation
  Output serialization: 5μs   (8%) ← pre-known structure
  Total:              ~60μs         ← vs 500μs for TEMPLATE (8x improvement)
```

## 3. What Gets Compiled

### 3.1 UTL-X AST Nodes → JVM Bytecode

| UTL-X Expression | AST Node | Generated Bytecode |
|-----------------|----------|-------------------|
| `$input.name` | PropertyAccess | `input.asObject().get("name")` — direct method call |
| `$input.items[0]` | IndexAccess | `input.asObject().get("items").asArray().get(0)` |
| `concat(a, b)` | FunctionCall | `StdlibString.concat(listOf(a, b))` — static call |
| `lowerCase(s)` | FunctionCall | `StdlibString.lowerCase(listOf(s))` — static call |
| `a + b` | BinaryOp(PLUS) | `RuntimeOps.add(a, b)` — static call |
| `a >= 18` | BinaryOp(GTE) | `RuntimeOps.gte(a, b)` — static call |
| `if (cond) a else b` | Conditional | `if (cond.isTruthy()) { a } else { b }` — JVM IFEQ |
| `let x = expr` | LetBinding | `val x = <expr>` — JVM local variable (ASTORE) |
| `{ key: value }` | ObjectLiteral | `UDM.Object(mapOf("key" to value))` |
| `[a, b, c]` | ArrayLiteral | `UDM.Array(listOf(a, b, c))` |
| `map(arr, (x) -> ...)` | FunctionCall + Lambda | Lambda compiled to inner class, `map` calls it |
| `filter(arr, (x) -> ...)` | FunctionCall + Lambda | Same pattern |
| `arr \|> map(...)` | Pipe | Desugared to `map(arr, ...)` before compilation |

### 3.2 What Stays Interpreted

| Concern | Why not compiled |
|---------|-----------------|
| Input parsing (JSON/XML/CSV/YAML) | Already optimized library code (Jackson, etc.) |
| Output serialization | Already optimized library code |
| Schema validation | Already uses pre-compiled validators (networknt, javax.xml) |
| Multi-input envelope splitting | Simple JSON parse, not worth bytecode |
| Format detection (auto mode) | One-time per message, trivial cost |

## 4. Architecture

### 4.1 Compilation Pipeline

```
.utlx source
    │
    ▼
Lexer → Tokens → Parser → AST (Program)
    │                          │
    │    (same as TEMPLATE)    │
    │                          ▼
    │                    Type Checker
    │                          │
    │                          ▼
    │                    ┌─────────────┐
    │                    │ AST Compiler │  ← NEW
    │                    └─────────────┘
    │                          │
    │                    Generates JVM bytecode
    │                    using ASM library
    │                          │
    │                          ▼
    │                    CompiledProgram
    │                    (Class<TransformFunction>)
    │                          │
    ▼                          ▼
TEMPLATE path            COMPILED path
(Interpreter.execute)    (TransformFunction.execute)
```

### 4.2 Generated Class

Each `.utlx` expression compiles to a class implementing a simple interface:

```kotlin
/**
 * Interface for compiled UTL-X transformations.
 * Generated at init-time by the AST compiler.
 */
interface TransformFunction {
    /**
     * Execute the compiled transformation.
     * @param inputs Named input UDMs (e.g., "input" → parsed JSON UDM)
     * @return The output UDM
     */
    fun execute(inputs: Map<String, UDM>): UDM
}
```

Example — for this UTL-X:
```
%utlx 1.0
input json
output json
---
{
  fullName: concat($input.firstName, " ", $input.lastName),
  email: lowerCase($input.email),
  isAdult: $input.age >= 18
}
```

The compiler generates (conceptually):
```kotlin
class CompiledTransform_a1b2c3 : TransformFunction {
    override fun execute(inputs: Map<String, UDM>): UDM {
        val input = inputs["input"]!!

        val firstName = input.asObject()!!.get("firstName")!!
        val lastName = input.asObject()!!.get("lastName")!!
        val email = input.asObject()!!.get("email")!!
        val age = input.asObject()!!.get("age")!!

        return UDM.Object(mapOf(
            "fullName" to StdlibString.concat(listOf(firstName, UDM.Scalar(" "), lastName)),
            "email" to StdlibString.lowerCase(listOf(email)),
            "isAdult" to RuntimeOps.gte(age, UDM.Scalar(18))
        ))
    }
}
```

But actually emitted as JVM bytecode via the ASM library — no source code generated.

### 4.3 CompiledStrategy Class

```kotlin
class CompiledStrategy : ExecutionStrategy {
    override val name = "COMPILED"

    private lateinit var compiledProgram: Program
    private lateinit var transformFunction: TransformFunction
    private val transformationService = TransformationService()

    override fun initialize(source: String, config: TransformConfig) {
        // 1. Compile .utlx → AST (same as TEMPLATE)
        compiledProgram = compileSource(source)

        // 2. Compile AST body → JVM bytecode → load class → instantiate
        val compiler = ASTCompiler()
        transformFunction = compiler.compile(compiledProgram)
    }

    override fun execute(input: String): ExecutionResult {
        // 1. Parse input (same as TEMPLATE — uses TransformationService)
        val inputUDM = parseInput(input)

        // 2. Execute compiled function (NO interpreter, direct method call)
        val outputUDM = transformFunction.execute(mapOf("input" to inputUDM))

        // 3. Serialize output (same as TEMPLATE)
        return ExecutionResult(output = serialize(outputUDM))
    }
}
```

### 4.4 ASTCompiler (the hard part)

```kotlin
class ASTCompiler {
    /**
     * Compile a Program's body expression to a TransformFunction class.
     * Uses ASM library to generate JVM bytecode directly.
     */
    fun compile(program: Program): TransformFunction {
        val className = "utlx/compiled/Transform_${hash(program)}"
        val classWriter = ClassWriter(ClassWriter.COMPUTE_FRAMES)

        // Generate class header
        classWriter.visit(V17, ACC_PUBLIC, className, null,
            "java/lang/Object", arrayOf("TransformFunction"))

        // Generate execute(Map<String, UDM>): UDM method
        val mv = classWriter.visitMethod(ACC_PUBLIC, "execute", "(...)")
        compileExpression(mv, program.body)  // recursive AST walk → bytecode
        mv.visitEnd()

        classWriter.visitEnd()
        val bytecode = classWriter.toByteArray()

        // Load the generated class and instantiate
        val clazz = BytecodeClassLoader().defineClass(className, bytecode)
        return clazz.getDeclaredConstructor().newInstance() as TransformFunction
    }

    private fun compileExpression(mv: MethodVisitor, expr: Expression) {
        when (expr) {
            is ObjectLiteral -> compileObjectLiteral(mv, expr)
            is PropertyAccess -> compilePropertyAccess(mv, expr)
            is FunctionCall -> compileFunctionCall(mv, expr)
            is BinaryOp -> compileBinaryOp(mv, expr)
            is Conditional -> compileConditional(mv, expr)
            is LetBinding -> compileLetBinding(mv, expr)
            is ArrayLiteral -> compileArrayLiteral(mv, expr)
            is Lambda -> compileLambda(mv, expr)
            is Pipe -> compilePipe(mv, expr)
            is StringLiteral -> compileStringLiteral(mv, expr)
            is NumberLiteral -> compileNumberLiteral(mv, expr)
            is BooleanLiteral -> compileBooleanLiteral(mv, expr)
            is NullLiteral -> compileNullLiteral(mv, expr)
            is VariableRef -> compileVariableRef(mv, expr)
            // ... all AST node types
        }
    }
}
```

## 5. Dependencies

| Library | Purpose | Size |
|---------|---------|------|
| `org.ow2.asm:asm:9.7` | JVM bytecode generation | ~400KB |
| `org.ow2.asm:asm-commons:9.7` | Helper utilities for ASM | ~80KB |

ASM is the standard for JVM bytecode generation — used by Kotlin itself, Spring, Hibernate, Gradle. Mature, fast, well-documented.

## 6. Scope and Complexity

### What makes this hard

1. **Every AST node type needs a bytecode emitter.** UTL-X has ~30 expression types. Each one needs correct stack management, type handling, and error paths.

2. **Lambda compilation.** Lambdas in `map()`, `filter()`, `reduce()` need to be compiled to inner classes or `invokedynamic` call sites. This is the most complex part.

3. **Stdlib function resolution.** 652 functions need to be resolvable at compile time as static method calls. Need a registry mapping function names to `MethodHandle` targets.

4. **Error handling.** Runtime errors (null access, type mismatch) must produce the same error messages as the interpreter. Bytecode needs try/catch blocks that wrap errors with UTL-X context.

5. **Testing.** Every compiled expression must produce identical output to the interpreter. Need a test harness that runs every conformance test through both TEMPLATE and COMPILED and compares results.

### What makes this tractable

1. **The AST is already well-structured.** Clean sealed classes, no ambiguous nodes.
2. **UDM is the only data type.** No need to generate code for multiple type systems — everything is `UDM` in, `UDM` out.
3. **Stdlib functions are already static methods.** Just need to look up the `@UTLXFunction` annotated methods and generate direct calls.
4. **The interpreter is the reference implementation.** Any bytecode disagreement = compiler bug. Easy to test.
5. **Incremental approach possible.** Start with a subset of AST nodes (literals, property access, object construction). Fall back to interpreter for unsupported nodes. Gradually expand coverage.

## 7. Implementation Approach: Incremental

### Phase 1: Core expressions (1-2 weeks)
- Literals (string, number, boolean, null)
- Object construction (`{ key: value }`)
- Array construction (`[a, b, c]`)
- Property access (`$input.name`, `$input.items[0]`)
- Variable references (`$input`)
- Binary operators (`+`, `-`, `*`, `/`, `==`, `!=`, `>=`, `<`, `&&`, `||`)
- Conditional (`if/else`)
- Let bindings

→ **Covers ~70% of real-world transformations**

### Phase 2: Functions + Lambdas (1-2 weeks)
- Direct stdlib function calls (`concat()`, `lowerCase()`, `upperCase()`, `size()`, etc.)
- Lambda expressions for `map()`, `filter()`, `reduce()`
- Pipe operator (`|>`)
- Null coalescing (`??`)
- Ternary (`?:`)

→ **Covers ~95% of real-world transformations**

### Phase 3: Edge cases + optimization (1 week)
- User-defined functions (PascalCase)
- Recursive descent (`..`)
- Wildcard access (`*`)
- String interpolation
- Error handling parity with interpreter
- JMH benchmarks vs TEMPLATE and COPY

### Fallback strategy
If a `.utlx` body contains an AST node the compiler doesn't support yet, fall back to the interpreter for that transformation. Log a warning: `"Expression type X not yet supported by COMPILED strategy, falling back to TEMPLATE"`. This allows incremental rollout.

## 8. Estimated Performance Impact

| Strategy | Init cost | Per-message cost (5KB JSON) | Throughput (8 workers) |
|----------|----------|----------------------------|----------------------|
| TEMPLATE | ~35ms (parse + AST) | ~500μs | ~24K msg/s |
| COPY | ~35ms + skeleton | ~300μs | ~44K msg/s |
| COMPILED | ~35ms + ~50ms (bytecode gen) | ~300μs | ~40K msg/s |
| COPY + COMPILED | ~35ms + skeleton + ~50ms | ~60μs | ~100K+ msg/s |

The biggest win is **COPY + COMPILED combined** — pre-built skeleton (no parsing) + compiled expressions (no interpretation). That's the Tibco-beating architecture.

## 9. Alternatives Considered

### Alternative A: Truffle/GraalVM partial evaluation
Use the Truffle framework to JIT-compile the AST automatically.
- **Pro:** Less manual work — Truffle handles the bytecode generation
- **Con:** Locks into GraalVM runtime, can't run on standard JVMs
- **Verdict:** Rejected — UTLXe must run on any JVM 17+

### Alternative B: Generate Kotlin source, compile at runtime
Generate `.kt` source code, invoke `kotlinc` at init-time.
- **Pro:** Human-readable generated code, easy to debug
- **Con:** Requires Kotlin compiler at runtime (~50MB), compilation takes seconds not milliseconds
- **Verdict:** Rejected — too slow for init-time, too large for deployment

### Alternative C: JavaScript/Nashorn compilation
Generate JavaScript, run on Nashorn/GraalJS.
- **Pro:** JavaScript JIT is very fast
- **Con:** Another runtime dependency, type conversion overhead between JVM and JS
- **Verdict:** Rejected — unnecessary complexity

### Alternative D: Direct ASM bytecode (recommended)
Generate JVM bytecode using ASM library.
- **Pro:** Fast generation (~50ms), fast execution, no extra runtime, standard approach
- **Con:** Complex to implement, hard to debug generated bytecode
- **Verdict:** Recommended — proven approach used by Kotlin, Groovy, Clojure compilers

---

## 10. Decision

**Expression compilation using ASM, incremental rollout, with interpreter fallback for unsupported nodes.**

Not scheduled for immediate implementation. This document serves as the design reference when the work begins.
